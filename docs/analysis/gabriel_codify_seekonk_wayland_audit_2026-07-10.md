## Whether boundary leakage recurred

**Yes, in a form the Massachusetts scale-up's fix did not yet anticipate — a real, useful finding.** This session's core deliverable (Task B) was fixing the *single-boundary* leakage pattern seen in the Massachusetts run (a returned excerpt crossing exactly one separator). That fix worked correctly on 3 of the 4 cases above. But the live run also surfaced a **multi-boundary** variant: `ma_seekonk_teacher_2021`'s no-strike excerpt spanned *six* separator instances in one continuous "verbatim" copy — the model badly over-captured across most of that window (likely because the teacher-contract window's repetitive table-of-contents-style text gave the model little signal about where a "genuine" span should end). The original single-split cleanup (keep the longer of two sides) still left several embedded `--- Excerpt N ---` fragments inside the "cleaned" text.

This was caught by manual review immediately after the live run (not a live-call failure — all 6 calls succeeded) and fixed **without any new API calls**: `_clean_boundary_leak()` in `scripts/gabriel_codify_pilot.py` was upgraded to segment on every complete separator instance and keep the single longest genuine segment (generalizing the two-way split to N-way), then `validated_outputs.csv` was regenerated by locally reprocessing the already-fetched `parsed_outputs.csv` through the corrected function. Verified against all 4 original Massachusetts single-boundary cases (still clean correctly) and the new multi-boundary case (now correctly isolates one genuine 280-character segment instead of a spliced ~1,400-character mess) before finalizing this audit.
