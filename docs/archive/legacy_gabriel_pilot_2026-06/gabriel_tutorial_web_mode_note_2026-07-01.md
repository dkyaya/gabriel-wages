# GABRIEL Tutorial Web-Mode Note

**Date:** 2026-07-01  
**Status:** local tutorial notebook not accessible in this session; note based on the tutorial correction supplied for this task and local project inspection

## Short note

The tutorial clarification changes the project framing materially. Built-in GABRIEL web mode should be treated as the primary live path, while the custom `get_all_responses_fn` scaffold should be treated as an advanced fallback.

## Supported tutorial paths to carry forward

### 1. `modality="web"` for `gabriel.rate`, `gabriel.classify`, and `gabriel.extract`

- The tutorial indicates these built-in functions can run in web mode by setting `modality="web"`.
- `search_context_size` controls how much retrieved web context is gathered for the prompt.
- Implication: GABRIEL can do direct web-grounded rating, classification, and extraction without first replacing the whole prompt-processing stack.

### 2. `gabriel.whatever(..., web_search=True)` as the report-first path

- The tutorial indicates a standard GABRIEL call can be asked to perform web search directly with `web_search=True`.
- `web_search_filters` narrow or prioritize the live search space.
- Implication: for this project, the first live test should be city-level web reports generated by built-in web search rather than a custom callback-first design.

### 3. `web_search_filters`

- The tutorial indicates `web_search_filters` are the built-in control surface for restricting or shaping search.
- In this project, that maps naturally to city-specific official-domain preferences such as municipal, school-district, union, and state-agency domains.
- Implication: our existing city prompt/domain-filter design is still useful, but it should likely feed built-in GABRIEL web mode first.

### 4. `search_context_size`

- The tutorial indicates `search_context_size` controls the amount of retrieved search context included for analysis.
- Implication: this is likely the first built-in knob to test when deciding whether report outputs are rich enough for downstream extraction.

### 5. `get_all_responses_fn` as advanced custom processing

- The tutorial clarification indicates `get_all_responses_fn` is an advanced hook for replacing the full prompt-processing stack.
- It is not the default route for standard GABRIEL web search.
- Implication: our `custom_get_all_responses` scaffold remains useful for schema control, custom batching, or nonstandard backends, but it should not be presented as the first-choice live mechanism.

## Project implications

- Primary live path should be: city prompt -> built-in GABRIEL web search/report generation.
- Likely extraction path should be: built-in web report -> `gabriel.extract` or structured parsing.
- The repo has not yet wired that built-in web mode into the project's city-by-city source and extraction schema.
- The current custom callback scaffold remains useful if built-in outputs are not structured enough, do not preserve the needed source/extraction schema, or require a nonstandard backend adapter.
- No live web search was run in this session.
- No GABRIEL model APIs were called in this session.
